## P0 (критично): выровнять `validate_fast` с `targets.csv`

### Проблема

Сейчас `validate_fast.py` формирует валидируемых пользователей так:

- `targets = incident_pos[["user_id"]].drop_duplicates()`

То есть оценка идет по **всем пользователям, у кого были события в октябре**, а не по `data/targets.csv`.

Фактические размеры:

- `targets.csv`: **3862** пользователей
- users с октябрьской активностью: **7569** пользователей

Это главная причина, почему ваш `validate_fast` NDCG заметно ниже платформенного:

- в локальную метрику попадает много "лишних" пользователей,
- средняя метрика на них тянет вниз итог.

### Что исправить

В `validate_fast.py`:

1. Брать базовый набор пользователей из `dataset.targets_df`.
2. Маскировать и считать recall только для пользователей из `targets.csv`.
3. В NDCG усреднять только по `targets.csv`.
4. Если у target-пользователя нет замаскированных пар в октябре, оставлять `NDCG=0` (это ок, но теперь это будет честно и сопоставимо).

### Ожидаемый эффект

- Метрика станет сопоставима с платформенной по пользовательскому срезу.
- Разрыв "0.009 vs 0.0127" частично сократится без изменения модели.

## P1 (важно): ограничить post-window явно ноябрем в ранкере

### Проблема

В `ranking.py` пост-инцидентный блок для affinity строится как:

- `positives[positives["event_ts"] >= _INCIDENT_END_TS]`

То есть верхняя граница (`< 2025-12-01`) не задана.

Сейчас датасет заканчивается ноябрем, поэтому ошибки нет. Но логика хрупкая: при расширении датасета будут втягиваться декабрь и дальше, что сломает интерпретацию "после инцидента = один месяц".

### Что исправить

Ввести `_POST_INCIDENT_END_TS = Timestamp("2025-12-01")` и в post-частях ранкера использовать интервал:

- `[_INCIDENT_END_TS, _POST_INCIDENT_END_TS)`

### Ожидаемый эффект

- Строгая привязка к постановке: инцидент в предпоследнем месяце, пост-сигнал только из последнего месяца.
- Устойчивость к будущим переупаковкам данных.

## P1 (важно): убрать "магическое +31 день" для конца инцидента

### Проблема

В `_item_popularity_features` (`ranking.py`) используется:

- `incident_end = _INCIDENT_START_TS + pd.Timedelta(days=31)`

Для октября 2025 это случайно корректно, но это неявная и хрупкая зависимость.

### Что исправить

Использовать константу `_INCIDENT_END_TS` напрямую везде, где нужен конец инцидента.

### Ожидаемый эффект

- Меньше шансов на тихие ошибки при переносе/рефакторинге.

## P2 (качество): сделать валидацию двухрежимной

### Проблема

Сейчас один `validate_fast` смешивает:

- strict-режим "по `targets.csv`",
- исследовательскую оценку "по всем октябрьским пользователям".

Это разные KPI для разных задач, их нельзя интерпретировать одинаково.

### Что исправить

Добавить флаг режима, например:

- `--eval-scope targets` (default)  
- `--eval-scope incident_users`

И печатать scope в итоге.

### Ожидаемый эффект

- Четкое разделение "боевой прокси-метрики" и "диагностической метрики".

## P3 (следующая итерация модели): приблизить псевдо-обучение к реальному инциденту

Это уже отмечено как P4 в плане и действительно остается TODO.

### Почему это важно

Сейчас YetiRank "слишком легко" учится на clean rolling windows (`bestTest=1.0`), что слабо переносится на реальный механизм потерь.

### Что делать

В `ranking.py` при построении pseudo-label окон добавить неравномерность, похожую на реальную:

1. Маскирование с зависимостью от user activity segment.
2. Разные вероятности для `event_type=1` и `event_type=2`.
3. Небольшую кластеризацию потерь по времени внутри окна.

### Ожидаемый эффект

- Более реалистичный train signal.
- Лучше перенос в октябрьский real incident.

## Как использовать факт "инцидент в предпоследнем месяце"

Это уже частично сделано правильно, но важно закрепить стратегию:

1. Октябрь (`2025-10`) — окно пропажи, значит кандидаты должны в первую очередь быть "октябрь-популярные" (у вас это есть).
2. Ноябрь (`2025-11`) использовать как **перенос вкуса пользователя**, а не как прямой item overlap (в EDA overlap почти нулевой).
3. Для target-пользователей без октябрьской активности, но с ноябрьской, делать упор на `post_incident_*` + CF (это отдельный сегмент с большим потенциалом).
4. Все offline-метрики считать на `targets.csv`, иначе нельзя корректно оптимизироваться под leaderboard.
